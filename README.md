# AIgnition 3.0 — Probabilistic Revenue Forecasting for E-commerce Marketing

**Submission for the NetElixir AIgnition 3.0 hackathon.**
An AI-assisted, fully offline forecasting utility that predicts e-commerce revenue and ROAS
from historical Google Ads, Microsoft (Bing) Ads, and Meta Ads data, at channel /
campaign-type / campaign grain, for 30/60/90-day planning horizons — with probabilistic
ranges (not single-point guesses) and plain-English causal summaries.

---

## 1. Quick start

```bash
# 1. Install dependencies (Python 3.11)
pip install -r requirements.txt

# 2. Run the graded pipeline end-to-end 
./run.sh ./data ./pickle/model.pkl ./output/predictions.csv

# 3. (Optional) Launch the interactive demo prototype
streamlit run app.py
```

Python version used for development & testing: **3.12** (also compatible with 3.10/3.11 per
pinned dependency versions in `requirements.txt`).

No internet access is required at run time — everything (model, code, sample data) is
committed to this repository.

## 4. AI integration — important design note

**As per an explicit project constraint for this submission, no
external or paid APIs are used anywhere in the prototype** — the entire pipeline runs with
zero network dependency and zero per-call cost, satisfying the submission guide's "no network
calls at run time" reproducibility rule by construction, not just by omission.

In place of a hosted LLM call, `src/insights.py` implements a deterministic **statistical
reasoning layer**: trend attribution, spend-elasticity interpretation, z-score anomaly
detection, and a template-based natural-language generator that turns those statistics into
plain-English, business-readable causal summaries and operational risk flags. This keeps the
*shape* of the required deliverable (causal explanation + anomaly interpretation, in plain
English) while remaining fully offline.

The function signature `segment_causal_summary(segment, weekly_history, model, forecast) -> str`
is intentionally structured so a real hosted LLM call could be
substituted behind the same interface later with no other code changes.

## 5. Data schema notes (for anyone replacing `data/` with new exports)

The pipeline auto-detects channel CSVs by filename pattern (see `src/utils.py` `LOADERS`) —
it does not hardcode file names. Expected raw schemas:

- **Google Ads** (`*google*.csv`): `campaign_id, segments_date, metrics_clicks,
  metrics_conversions, metrics_cost_micros, metrics_impressions, metrics_conversions_value,
  campaign_advertising_channel_type, campaign_budget_amount, campaign_name`
- **Microsoft/Bing Ads** (`*bing*.csv`): `CampaignId, TimePeriod, Revenue, Spend, Clicks,
  Impressions, Conversions, CampaignType, DailyBudget, CampaignName`
- **Meta Ads** (`*meta*.csv`): `campaign_id, date_start, cpc, cpm, ctr, reach, spend, clicks,
  impressions, conversion, daily_budget, campaign_name`. Meta's `conversion` field is treated
  as a conversion **value** (revenue), consistent with its scale relative to spend in the
  supplied sample data; Meta does not expose a native campaign-type field, so one is inferred
  from the campaign-naming convention (Prospecting / Remarketing / Brand / Catalog-DPA /
  Generic — see `infer_meta_campaign_type()`).
